깃허브 이슈(Github Issues)
: GitHub Issues는 프로젝트에서 생기는 할 일, 버그, 개선 요청을 기록·관리하기 위한 공식적인 작업 관리 도구.
따라서 팀원 모두가 같은 공간에서 문제를 공유하고 해결 과정을 추적할 수 있게 만들어주었습니다.

① 작업 관리 기능성, ② 협업 효율성, ③ 유지·추적 관리 품질 부분에서 더 용이하게 사용하게 되었습니다.
따라서 업무 관리에 큰 도움이 되었지만, 몇 가지 한계점과 개선이 필요한 부분도 확인되었습니다.

아래는 GitHub Issues 활용 과정에서 나타난 용이한점들과 주요 문제점, 그리고 향후 개선 방향을 정리한 내용입니다.


1. 작업 관리 기능성
● 작업 단위 명확화: 버그, 기능 추가, 문서 작업 등을 개별 이슈로 분리하여 업무 구조가 보다 명확해졌습니다.
● 레이블·마일스톤 활용: 중요도 및 일정 관리가 가능해져 작업 우선순위 파악이 용이해졌습니다.
● 이슈 템플릿 활용: 일정한 형식으로 이슈를 작성하게 되어 정보 누락이 줄고 재현 가능성이 높아졌습니다.

2. 협업 효율성
● 담당자 지정기능: 작업 책임자가 명확해져 팀 내 역할 분담이 원활해졌습니다.
● 댓글 기반 논의 활성화: 이슈 내에서 모든 논의를 진행할 수 있어 외부 메신저 없이도 소통이 효율적으로 이루어졌습니다.
● PR과 이슈 연동: Pull Request와 연결하여 어떤 변경이 어떤 이슈를 해결했는지 쉽게 추적할 수 있었습니다.

3. 유지·추적 관리 품질
● 상태 변화 추적: Open → In Progress → Closed 형태로 상태를 관리하며 프로젝트 진행 상황을 명확하게 확인할 수 있었습니다.
● 해결 과정 기록 유지: 수정 과정, 논의 내용 등이 자동으로 누적되어 향후 참고 자료로 활용하기 수월했습니다.
● 중복·방치 이슈 감소: 검색과 필터 기능을 통해 기존 이슈 확인이 쉬워져 중복 이슈나 미해결 이슈가 줄어들었습니다.

4. 한계점
● 이슈 작성 방식의 일관성 부족: 팀원마다 작성 형식이 달라 정보 누락이 발생하거나, 제목·설명이 모호하여 문제 파악에 시간이 소요되는 경우가 있었습니다.
● 상태 업데이트 지연: Open, In Progress, Closed 상태 변경이 즉시 이루어지지 않아 실제 진행 상황과 표시된 상태가 일치하지 않는 문제가 발생하였습니다.

5. 개선이 필요한 부분
● 이슈 템플릿 강화: 재현 단계, 기대 결과, 실제 결과 등 필수 항목을 포함한 템플릿을 적용하여 이슈 문서화의 일관성과 재현 가능성을 높일 필요가 있습니다.
● 우선순위 기준 명확화: 레이블 체계를 정비하고 정기적인 이슈 리뷰 시간을 도입하여 작업 분배 효율성을 높이고 방치된 이슈를 줄일 필요가 있습니다.
