# GitHub 마일스톤(Milestone) 기능 평가
    1. 개요

        GitHub 마일스톤은 여러 이슈(issue)와 풀 리퀘스트(pull request)를 하나의 목표 단위로 묶어 관리하는 기능이다.
        기능 구현, 버그 수정, 릴리스 등 프로젝트의 주요 구간을 마일스톤 단위로 나눔으로써, 프로젝트의 전체 흐름과 현재 위치를 파악하는 데 도움을 준다.
        마감일(Due date)을 설정할 수 있어 일정 기반 개발 계획 수립에도 활용할 수 있다.

    2. 마일스톤의 주요 장점

        2.1 프로젝트의 큰 그림 관리

            여러 이슈와 PR을 하나의 스프린트·릴리스·기능 단위로 묶어 "이정표"처럼 관리할 수 있다.
            버전, 베타/정식 릴리스, 기능 묶음, 주 단위 스프린트 등으로 마일스톤을 설정하면 프로젝트 전체 로드맵이 구조화된다.

        2.2 진행 상태의 가시화와 추적

            마일스톤별로 연결된 이슈·PR의 완료 여부가 집계되어, 전체 진행률을 퍼센트로 확인할 수 있다.
            마감일을 설정하면 남은 기간과 남은 작업량을 함께 고려할 수 있어 일정 관리에 유리하다.
            마일스톤 화면에서 열린 이슈/PR과 닫힌 이슈/PR을 나누어 볼 수 있어, 현재 상태를 빠르게 파악할 수 있다.

        2.3 이슈 분류 체계와 협업 워크플로우 단순화

            이슈에 라벨을 붙이고 마일스톤을 지정하면, "라벨 + 마일스톤"이라는 2중 구조로 작업을 정리할 수 있다.
            여러 기능, 모듈, 릴리스가 동시에 진행되는 경우에도 마일스톤 단위로 그룹을 나누어 진행률과 데드라인을 관리할 수 있다.

        2.4 다른 기능과의 통합 사용

            마일스톤은 GitHub Projects(칸반 보드), 라벨, 이슈 템플릿 등과 함께 사용할 때 효과가 커진다.
            이슈뿐 아니라 PR도 마일스톤에 연결할 수 있어, 기능 개발, 코드 병합, 릴리스까지의 흐름을 하나의 목표 단위 안에서 관리할 수 있다.
            3.5 별도 툴 없이 제공되는 기본 프로젝트 관리 도구
            JIRA, Asana, Trello 같은 별도 툴 없이도 GitHub만으로 이슈 관리, 릴리스 계획, 진행 상황 추적을 어느 정도 해결할 수 있다.
            인력이 적거나 리소스가 제한된 오픈소스, 개인, 소규모 프로젝트에서 부담 없이 사용 가능하다.

    3. 한계점과 주의사항

        3.1 복잡한 워크플로우나 세밀한 태스크에는 부족할 수 있음

            마일스톤은 원래 큰 단위 목표(릴리스, 스프린트, 기능 묶음) 관리를 위해 설계되었다.
            세밀한 태스크 단위나 자주 변하는 작은 작업까지 모두 마일스톤으로 관리하려 하면 오히려 혼란이 생길 수 있다.
            세부 태스크 추적과 우선순위 관리는 프로젝트 보드, 라벨, 이슈 템플릿 등을 활용하는 편이 더 적합하다.

        3.2 하나의 이슈는 하나의 마일스톤만 가질 수 있음

            한 이슈는 동시에 여러 마일스톤에 속할 수 없다는 제약이 있다.
            같은 작업이 여러 목표(예: 기능 릴리스와 버그 수정 릴리스)와 연관될 경우 설계가 제한될 수 있어, 마일스톤 기준을 명확히 정해둘 필요가 있다.
            보통 릴리스 기준 또는 기능 기준 중 하나를 우선으로 정해서 사용한다.

        3.3 작은 프로젝트에서는 과한 구조가 될 수 있음

            이슈 수가 적고 단순한 개인 프로젝트에서는 마일스톤을 만드는 것 자체가 관리 비용이 될 수 있다.
            이런 경우에는 이슈 + 라벨 + 간단한 보드만으로도 충분한 경우가 많다.

        3.4 Projects와 함께 사용할 때 복잡도가 증가할 수 있음

            Project 보드와 마일스톤을 동시에 운영하면 초기 설정과 운영 방식이 복잡하게 느껴질 수 있다.
            팀마다 마일스톤 사용 규칙(기능 기준, 릴리스 기준, 스프린트 기준 등)이 다르면, 오히려 관리 오버헤드가 증가할 수 있다.
            일관된 정책과 최소한의 규칙 정리가 필요하다.

    4. 종합 평가

        GitHub 마일스톤은 이슈와 PR 중심의 개발 흐름을 릴리스, 스프린트, 기능 단위 목표로 구조화하고, 진행 상태를 시각적으로 보여주는 실용적인 기능이다. 별도 도구 없이도 일정 관리와 목표 관리의 기본 틀을 제공한다는 점에서 가치가 크다.

        다만, 효과는 마일스톤을 어떻게 설계하고 다른 기능(라벨, 프로젝트 보드 등)과 어떻게 조합해 사용하는지에 따라 달라진다.
        프로젝트가 단순하거나 이슈가 적은 경우에는 불필요한 복잡성을 추가할 수 있고, 프로젝트 규모가 크거나 협업 인원이 많고 릴리스가 반복되는 환경에서는 거의 필수적인 관리 수단이 된다.
