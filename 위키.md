# 깃허브(GitHub) 위키 기능 평가 보고서

---

## 1. 개요 (Executive Summary)
본 보고서는 GitHub 리포지토리(Repository)에 내장된 위키(Wiki) 기능의 효용성, 장단점 및 적합한 사용 사례를 분석합니다.

GitHub 위키는 **개발자 친화적인 문서화 도구**로서 코드와 문서의 근접성을 높이는 데 탁월하지만, 복잡한 계층 구조 관리나 비개발 직군과의 협업에는 명확한 한계가 존재합니다. 따라서 프로젝트의 규모와 팀의 성격에 따라 도입 여부를 신중히 결정해야 합니다.

---

## 2. 주요 기능 및 장점 (Pros)

### 2.1. 코드와의 높은 접근성 (Proximity)
* **통합 환경:** 코드가 저장된 리포지토리 내 탭(Tab)으로 존재하여, 개발자가 별도의 외부 툴(Notion, Confluence 등)로 이동하지 않고 문서를 열람하거나 작성할 수 있습니다.
* **문맥 유지:** 프로젝트의 README.md보다 방대한 양의 정보를 담을 수 있어, API 문서나 아키텍처 설명을 코드 바로 옆에 배치하기 좋습니다.

### 2.2. Git 기반 버전 관리 (Version Control)
* **자체 리포지토리:** 위키 자체가 하나의 독립적인 Git 리포지토리입니다. 따라서 문서의 모든 변경 사항(History)이 커밋(Commit) 단위로 기록되며, 언제든 이전 버전으로 롤백이 가능합니다.
* **로컬 편집:** 웹 인터페이스뿐만 아니라, 위키 리포지토리를 로컬에 `clone`하여 좋아하는 에디터(VS Code 등)로 작성 후 `push`할 수 있습니다.

### 2.3. 표준 마크다운(Markdown) 지원
* 개발자에게 익숙한 Markdown 문법을 기본으로 지원하여, 코드 블록 하이라이팅, 링크 삽입 등이 용이합니다.

---

## 3. 주요 한계점 (Cons)

### 3.1. 빈약한 검색 및 구조화 기능
* **검색의 한계:** 위키 내 검색 기능이 존재하지만, 단순 키워드 매칭 수준이며 오타 보정이나 시맨틱 검색이 지원되지 않습니다.
* **계층 구조의 부재:** 기본적으로 'Flat'한 구조를 가집니다. 사이드바(Sidebar)를 커스텀하여 계층을 흉내 낼 수는 있으나, Notion이나 Confluence처럼 드래그 앤 드롭으로 문서를 정리하는 직관적인 UX는 제공하지 않습니다.

### 3.2. 실시간 협업 불가
* 동시 편집 기능이 없습니다. 두 명이 동시에 같은 문서를 수정하고 저장하려 할 경우, Git 충돌(Conflict)이 발생하거나 나중에 저장한 사람의 내용이 덮어씌워질 위험이 있습니다.

### 3.3. 비개발 직군 진입 장벽
* 이미지를 붙여넣기(Paste) 할 때 자동으로 업로드되지 않는 경우가 많으며(별도 이슈 트래커에 올려서 URL을 따오거나 리포지토리에 올려야 함), 위지윅(WYSIWYG) 에디터가 빈약하여 기획자나 디자이너가 사용하기에 불편합니다.

## 4. 경쟁 도구 비교 (Comparison)

| 기능 | GitHub Wiki | Notion | Confluence |
| :--- | :--- | :--- | :--- |
| **타겟 사용자** | 개발자 | 전사 직원 (기획/디자인/개발) | 엔터프라이즈 팀 |
| **작성 방식** | Markdown (Git 기반) | 블록형 에디터 (WYSIWYG) | Rich Text 에디터 |
| **버전 관리** | 완벽함 (Git History) | 페이지 히스토리 (제한적) | 상세 히스토리 제공 |
| **구조화** | 수동 관리 (Sidebar) | 매우 유연함 (트리 구조) | 유연함 (트리 구조) |
| **비용** | 무료 (Public/Private) | 유료 (팀 단위 과금) | 유료 (사용자 수 과금) |

---

## 5. 종합 평가 및 추천 (Verdict)

### ✅ 추천하는 경우 (Use Cases)
1. **오픈소스 프로젝트:** 기여자(Contributor)들이 코드와 문서를 한곳에서 보고 관리해야 할 때 표준적으로 사용됩니다.
2. **소규모 개발팀:** 복잡한 기능 명세보다는 API 사용법, 서버 세팅 가이드 등 **"개발자를 위한 개발 문서"**를 작성할 때 가장 효율적입니다.
3. **비용 절감:** 별도의 문서화 도구 비용을 지출하고 싶지 않을 때 훌륭한 대안입니다.

### ⛔ 비추천하는 경우
1. **기획/디자인 팀과 협업:** 화면 설계서나 기획서를 공유하기에는 이미지 관리와 댓글 기능이 너무 부족합니다.
2. **복잡한 사내 지식 베이스:** 검색 기능이 약하고 카테고리 분류가 어려워, 문서가 100개를 넘어가면 관리가 불가능해집니다.
